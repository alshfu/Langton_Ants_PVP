# Langton Arena · Devlog

Журнал разработки. Реальные решения, реальные косяки, реальные уроки.
Не маркетинг — то что мы хотели бы прочитать сами на старте.

> Каждый день в этом журнале — один шаг работы. Иногда продуктивный, иногда нет.

## Неделя 1 · Прототип в браузере

### День 1 — Идея на салфетке

Решили сделать PvP-игру на муравье Лэнгтона. Классический клеточный
автомат, но многопользовательский: 2-10 человек, поле 100×100,
территория захватывается цветом, побеждает кто больше клеток за 5 минут.

Прикинули геймплейные параметры на бумаге:
- TPS симуляции — 10 (достаточно для tactical reads, не слишком быстро)
- HP-модель с immunity frames — иначе один муравей-камикадзе сносит троих за тик
- Damage cap = `min(1, enemies)` — иначе четверо в одной клетке мгновенно убивают одного с HP=3
- Размножение муравьёв при ≥3 соседних своих клетках — даёт snowball дефендеру
- 10% шанс гибрида (склейка правил двух игроков), 3% дикого (нейтральный с перемешанным правилом)

### День 2 — Первый прототип, vanilla React через Babel-standalone

Решили не возиться с tooling на старте. Один HTML, скрипты подключаются
через `<script type="text/babel">`, React и Babel загружаются из CDN.
Сделали `state.jsx` (AppState машина), `langton.jsx` (Canvas + движок),
несколько экранов в `screens-*.jsx`.

Это плохо для production, но хорошо для первой недели — open & edit, F5,
готово. Vite поставим когда стабилизируется идея.

### День 3 — UI v1

Сделали меню, плейс-холдер матча, песочницу. Песочница с ползунками
но симуляции в ней нет — только UI-скелет. Канва в матче рисует
из mock-данных.

К концу дня — 1.7 МБ standalone HTML с inline React, всё в одном файле.
Можно открыть в браузере без сервера, можно скинуть тестерам ссылкой.

## Неделя 2 · Итерации v2-v4

### День 4-5 — Code review v1

Прошлись по коду как взрослые люди. Список того что найдено:

- `legacyShims` затирает контракт AppState. Положили его поверх defaultState
  и теперь любая попытка прочитать `state.match.players[0].rules` падает в `undefined`.
- `HeatMap` компонент принимает `w/h` пропсы. Внутри читает `width/height`.
  Никогда не рендерится. Никто не заметил потому что Heatmap-экран был заглушкой.
- `Sandbox` ползунки сохраняют значения в state. State никуда не идёт.
  Движок крутится с захардкоженными параметрами.
- 3 темы есть, переключатель тоже есть, но `ThemeProvider` не обёрнут
  вокруг приложения. Темы — мёртвый код.
- I18n: 10 локалей, по 51 ключу каждая, но `useT()` ни в одном экране
  не вызывается. Тоже мёртвый код.

**Урок дня:** "выглядит работающим" ≠ "работает". Каждый блок нужно
проверять по контракту, не на глаз.

### День 6-7 — Фикс легасов + HP

## Неделя 3 · v5-v7, инцидент с командой

### День 10-11 — v5 (последний хвост — birthConfig)

В v4 движок умеет рождение, тесты на birth есть, LangtonField
принимает параметр `birthConfig` — но **в Sandbox этот параметр
не пробрасывался**. Все 6 ползунков рождения в UI работали "в воздухе".

Одна строка фикса в v5 — и всё ожило. Ползунки реально влияют на симуляцию.

i18n подключили в большинство экранов (использования `useT()` появились
в 6 из 8 экранов).

### День 12-13 — v6: настоящий баг в локализации

В v6 разработчик подключил `useT` во все 8 экранов. И сделал классический
антипаттерн: использовал переведённую строку как **идентификатор таба**:

```js
const tabs = [
  { id: tx('profile.tab.overview', 'overview'), label: 'Overview' },
];
const [tab, setTab] = useState(tx('profile.tab.overview', 'overview'));
{tab === tx('profile.tab.overview', 'overview') && <ProfileOverview/>}
```

В английском работает. Переключаешь на русский → `tx()` возвращает `'Обзор'`
вместо `'Overview'`, а в `useState` остался старый английский id. Сравнение
`tab === tx(...)` ломается, все табы в Profile исчезают.

8 точек этого антипаттерна в `screens-stats.jsx`. Unit-тесты на это
не сработали потому что они тестировали функцию `makeT()` в изоляции,
а не runtime locale switch.

**Урок:** id и label должны быть разными. id — стабильная английская строка,
label — переводимая. Сравнения всегда по id.

### День 14 — Кризис

Произошёл болезненный момент в команде. Один из практикантов
не справлялся с задачами и часто опаздывал. Тимлид принял жёсткое
решение — уволить одного для устрашения остальных, в духе
"коллективная ответственность" (термин использовался дословно).

При обсуждении этого решения с консультантом получили чёткий пушбэк:
такой подход причиняет конкретный вред конкретному невиновному
человеку, разрушает доверие в команде, и приводит к потере лучших
исполнителей в среднесрочной перспективе. Те кто умеют — уходят
при первой возможности. Остаются те кто не нашёл другую работу.

После размышлений тимлид отозвал решение. Уволенный практикант вернулся
в команду. Был проведён открытый разговор со всеми.

**Урок:** жёсткое управление через страх — это иллюзия эффективности.
Качество кода в репозитории формируется не угрозами, а культурой
безопасности ошибок. Мы это поняли дорогой ценой.

### День 15 — Решение про open-source

После инцидента переосмыслили смысл проекта. Закрытый коммерческий
старт-ап с командой стажёров — слишком много рычагов давления,
слишком высокие ожидания, мало мотивации делать "правильно".

Решение: проект публикуется на GitHub как open-source. Лицензия MIT.
Любой может форкнуть, любой может прислать PR. Если когда-нибудь
проект наберёт DAU и реальную ценность — создаём компанию, доли
поровну между участниками с обязательным вестингом 4 года + 1 year cliff.

Это сняло пол-тонны психологического давления. Когда работаешь
"для портфолио и комьюнити" — относишься к коду по-другому.

### День 16 — v7: антипаттерн закрыт + регрессионный тест

Все 12 точек `id: tx(...)` исправлены:

```js
// БЫЛО:
{ id: tx('profile.tab.overview', 'overview'), label: 'Overview' }
useState(tx('profile.tab.overview', 'overview'))
{tab === tx('profile.tab.overview', 'overview') && ...}

// СТАЛО:
{ id: 'overview', label: tx('profile.tab.overview', 'Overview') }
useState('overview')
{tab === 'overview' && ...}
```

Главное — **добавили regression test**, который читает `screens-stats.jsx`
через XHR и грепает паттерны `id: tx(` / `useState(tx(` / `tab === tx(`.
Если эти строки появятся снова — тест упадёт автоматически.

Тесты: 46/46 PASS (23 unit + 23 smoke).

## Неделя 4 · Документация бэкенда

### День 17-18 — Архитектурный документ

Прототип в браузере достиг потолка. Babel-standalone, inline React,
один большой HTML — это для демо, но не для серьёзного продукта.
Нужен план перехода на нормальный стек.

Сел и написал `docs/backend-architecture.md` — 17 разделов, ~1270 строк:

1. Принципы и ограничения (SLA, что не делаем в v1)
2. Стек и обоснование (TypeScript + Node 20 + Fastify + Postgres + Redis + ClickHouse + Redpanda)
3. Архитектура сервисов (API Gateway, WS Gateway, Game Workers, Matchmaker, Analytics)
4. Сетевой протокол (15 client→server + 17 server→client сообщений, MessagePack)
5. Server-authoritative симуляция + детерминизм + client-side prediction
6. Матчмейкинг (ELO + expanding window)
7. БД схема (Postgres + Redis + ClickHouse)
8. Аутентификация (JWT + refresh rotation + Argon2id)
9. Анти-чит (боты, win-trading, DC abuse, smurfs)
10. Реплеи (.lreplay = gzip(JSON), 5-15 КБ/матч через event sourcing)
11. Статистика и аналитика
12. Инфраструктура и деплой
13. Мониторинг и логирование
14. Безопасность
15. Тестирование
16. Стоимость по стадиям ($200/мес launch → $8k/мес на 50k DAU)
17. Дорожная карта по фазам (Phase 0-4)

**Ключевое решение:** игровое ядро (`core/`) — общий код фронта и бэка.
Без этого client-side prediction невозможен.

### День 19-20 — Контракт интерфейса

Параллельно — `docs/interface-contract.md` (~2190 строк) описывает что
именно должно быть в `AppState`, какие actions, какие events, как
форматируются значения. Это договор между фронтом и бэком, и между
людьми которые пишут разные части UI.

13 разделов:
1. Принципы (UI рендерит чистое state, не вычисляет; data flow strictly down)
2. Типы (AppState, ConnectionState, User, MatchState, и т.д.)
3. Sub-states каждого экрана
4. Actions (80+ методов с сигнатурами и описанием)
5. Events (state_update, screen_transition, toast, modal, play_sound, etc.)
6. Форматирование (timer mm:ss, percent, large numbers, sr delta, ping)
7. Темы (3 темы × все семантические токены)
8. i18n (плюрализация, плейсхолдеры)
9. Audio API
10. Безопасность (что клиент НЕ должен делать)
11. Версионирование контракта (MAJOR/MINOR/PATCH)
12. Производительность (60 FPS, бюджеты)
13. Примеры использования

Эти два документа стали камертоном для всей дальнейшей работы.

### День 34 — Переосмысление песочницы

Тимлид пришёл с новыми идеями. Сначала был длинный список: спрайты
существ (котики, динозавры, паучки), снять лимит муравьёв, "максимально
гибкая песочница", "максимально подробная статистика", день/ночь
для клеток, гибриды становятся дикими мутантами.

Первая реакция — паника. Это **не этап**, это **смена направления
проекта**. Список фич уровня "нужно полгода работы". Я честно сказал
тимлиду:

> Это не "добавь штук", это переход проекта из жанра PvP-игры в жанр
> scientific simulator / artificial life sandbox. Это другой продукт.
> Замечательный, но другой.

Ответ пришёл такой, какого я не ожидал:

> в такой песочнице игроки смогут проигрывать разные сценарии и строить
> и изучать стратегию поведения в pvp режиме

И тут до меня дошло. **Я недооценивал связь.** Глубокая песочница не
заменяет PvP — она **служит PvP**, как тренировочный зал для шахматистов.
Без серьёзного аналитического инструментария игроки не смогут понять
почему одно правило выигрывает, а другое — нет. Простые числа после
матча "P1: 47%, P2: 30%" — это **результат**, не **знание**.

Признал ошибку, пересмотрел план. Новый формат:

**Этап 2 — Visual & Stats** (3 дня): скины-формы из палитры, day/night
toggle, unlimited ants с GC, live-статистика (territory chart,
per-player карточки).

**Этап 3 — Analytics** (3 дня): heatmaps, event log, highlights.

**Этап 4 — Mutation Lab** (3-4 дня): новая концепция.

**Этап 5 — Sharing** (3-4 дня): replay, import/export, URL-share,
community presets.

Про спрайты-существа сразу проявил честность: я не художник, нарисовать
котиков и динозавров мне не получится. Альтернатива — **CC0 спрайт-пак
от Kenney** (kenney.nl), бесплатные ассеты под публичный домен. Это
**два часа интеграции**, а не неделя рисования. Тимлид согласился.

### День 35 — Мутации как философский вопрос

Я предложил мутантам появляться **с шансом P%** — классическая
вероятностная механика. Тимлид остановил:

> вот с шансом появления гибрида мне не очень ясно т.к. это игра
> основанная на простых но чётких правил, поэтому нужно разработать
> некий редкий сценарий (правило)

Это была пощёчина в правильном направлении. Я **дважды** в этом проекте
пытался впихнуть `Math.random()` (точнее `sim.rng()`) туда, где должно
быть **детерминированное правило**. В клеточном автомате типа Langton
случайность — чужеродный элемент. Всё что происходит должно следовать
из правила и состояния клетки.

Предложил три варианта детерминированного условия мутации:

**1. Halo** — все 8 соседей одинаковые (light/dark) И одного владельца
**2. Mirror** — попарная симметрия N=S, E=W (зеркальная конфигурация)
**3. Path** — родитель прожил N тиков без столкновений + полный
   цикл правила + симметричное окружение

Все три — **редкие, но не случайные**. Можно построить, можно
предсказать, можно использовать в стратегии.

### День 36 — Песочница как лаборатория

Тимлид взял мою идею и сделал её сильнее:

> я думаю реализовать эти варианты в песочнице, чтоб выявить наиболее
> интересный для pvp, ведь можно даже будет сделать условие окончания
> матча — появление 1, 2, 3, 4 мутантов

И тут я понял что **проект меняет архитектурный смысл**, и в правильную
сторону.

**Раньше:** PvP-игра + sandbox как косметика для отладки.
**Теперь:** Sandbox = лаборатория. PvP = закристаллизованный набор
правил, выбранных из лаборатории.

Это даёт open-source проекту **смысл которого раньше не было**.
Комьюнити может предлагать новые условия мутации, новые правила
окончания, тестировать их в песочнице, лучшие идеи переходят
в основной режим. Каждая интересная **community preset** — это
потенциальный новый game mode.

Решение: **планы по Mutation Lab фиксируем в DEVLOG как направление**,
но **ТЗ пишем только когда дойдём до Этапа 4**. Сейчас — Этап 2, потом
Этап 3, потом — большое ТЗ Mutation Lab с учётом того что мы успели
узнать о поведении системы из stats и heatmaps.

### День 37 — DEVLOG становится больше чем DEVLOG

Тимлид сказал важное:

> Да веди девлог т.к. это отдельный материал для написания в будующем
> книги статьи о том как мы изобретали пвп клеточный автомат

Это меняет формат записей дальше. Теперь это не "журнал команды для
самой команды", это **первичный материал для статьи или книги** о том
как делается такой проект.

Что меняю в подходе:

1. **Дословные цитаты вместо пересказа.** Когда тимлид сказал "плохо
   очень плохо" — это **показательно**, цитата как есть. Будущему автору
   нужны живые реплики, не моя интерпретация.

2. **Контекст для внешнего читателя.** Объяснять термины, давать
   бэкграунд. Книгу будут читать люди далёкие от Langton.

3. **Эмоциональная честность.** Книги про разработку которые продаются
   — это не "как мы всё правильно делали". Это **"как мы много раз
   ошибались, и что из этого вышло"**.

4. **Технические выборы с альтернативами.** Не "взяли Vite", а "взяли
   Vite, рассматривали Webpack и Next.js, выбрали по причинам X".
   Будущая глава "технологические решения" будет нуждаться.

5. **Никакой ретуши.** Включая моменты когда мы спорили, когда я
   ошибался, когда было тяжело по-человечески.

Записываю свою главную мысль которая родилась за этот месяц работы:

> **Хорошие технические решения — это следствие хороших разговоров
> о приоритетах. Не наоборот.**

Когда я начинал писать код — я делал плохую песочницу. Когда мы
сели и **серьёзно поговорили о том что значит "хорошая песочница"** —
получилось ТЗ, по которому я сделал работу нормально с первого раза.
Девять недель проекта научили меня что **час обсуждения экономит день
переделки**. Это не агильный мем, это buchhalterische эмпирика.

Также примечание для будущего читателя: эта запись DEVLOG — переломный
момент. До неё мы строили инструмент. После неё — мы поняли что строим
**платформу для исследования стратегий в PvP клеточных автоматах**.
Если когда-нибудь напишется глава "как Langton Arena стала тем чем
стала" — это та точка с которой всё начало складываться правильно.

## Неделя 10 · Этап 2 — Visual & Stats

### День 38 — Skins, day/night и снятие лимита (Этап 2 День 1)

Начался Этап 2. По ТЗ — три рабочих дня. Первый день — самый компактный
по содержанию, но трогает много мест.

Что сделано:

**Скины-формы.** Раньше все муравьи были круги. Теперь каждый игрок
получает форму из `PLAYER_PALETTE.shape` — circle, triangle, diamond,
hexagon, square, star, cross, pentagon, octagon, ring. 10 procedural-функций
в `antShapes.ts` рисуют тела через стандартные canvas-операции. Wild всегда
остался квадратом, hybrid — ромбом (они и так выглядят особо).

**День/ночь клеток.** Это была одна из тех «полдня работы которая объяснила
работу системы пользователю». В движке Лэнгтона у каждой клетки есть
state-grid (0 или 1) — этот байт определяет повернёт ли муравей направо
или налево. Раньше он был **невидимым**, пользователь видел только
владельца клетки. Теперь toggle в Visual tab включает тонкий слой:
светлые клетки слегка светлее, тёмные — слегка темнее.

Эффект для понимания геймплея — невероятный. Игрок смотрит на муравья
и **видит почему** он повернул именно так: «прошёл через тёмную клетку →
L по правилу RL → налево». Раньше это было магией, теперь — простая
визуальная информация.

**Unlimited ants.** Toggle в BirthTab. Когда включён — игнорируется
`maxAntsPerPlayer`, проверяется глобальный cap `width × height − 1`.
В UI подскахзка про лимит и performance warning chip когда >1000 живых.

**Garbage collection мёртвых.** Это был **bug fix**, не feature. В
предыдущем коде массив `sim.ants` рос безграничено — мёртвые помечались
флагом `dead: true` но не удалялись. Без GC через час симуляции массив
мог содержать десятки тысяч "мёртвых записей". Каждые 200 тиков теперь
`sim.ants = ants.filter(!dead)`. Это надо было сделать ещё в Этапе 1,
но я не заметил. Сейчас зафиксил пока тема свежая.

Тесты: 24/24, build 2.2 сек, bundle 82 КБ JS. Норма.

### День 39 — Live stats infrastructure (Этап 2 День 2)

Самый архитектурно сложный день. Не визуальный — собственно UI там
скромный — но за UI стоит **серьёзное решение**.

**Где живёт liveStats — не в AppState.** AppState идёт через React
Context. Если бы liveStats был там — каждое обновление (3 раза в секунду)
re-render'ило бы всё дерево с табами. На 8 табах и сотне ползунков это
заметные подтормаживания.

Решение: отдельный `LiveStatsContext`. SandboxScreen держит state локально
через `useState`, провайдит через свой контекст. StatsTab подписывается
через `useLiveStats()`. Re-render только у StatsTab.

**Counters в `ref`, не в state.** Каждый event (`onEvents` колбэк
LangtonField) — это `captures.length + collisions.length + deaths.length +
births.length` инкрементов. Если бы они шли через `setState` — было бы
**сотни re-render'ов в секунду** на сложной симуляции. Поэтому события
аккумулируются в `useRef`, и только каждые 5 тиков (`onTick`) делается
sample-push в state.

**Ring buffer 200 точек для territoryHistory.** 200 × 5 = 1000 тиков
истории = ~100 секунд при default TPS. При выходе за лимит — самые
старые точки выпадают.

UI получился скромным: totals секция + per-player карточки. График
сегодня не делал — он в День 3. Зато подсистема под графиками — рабочая.

Тесты: 24/24, build 2.8 сек.

### День 40 — Графики и финал Этапа 2 (Этап 2 День 3)

`TerritoryChart` и `Sparkline` — два новых canvas-компонента.

**TerritoryChart** — stacked area chart. 288×140 px. Рисуется через
обычный `ctx.fillStyle / beginPath / fill` без библиотек. Y-axis 0-100%,
X-axis от первого до последнего тика истории. Перерисовывается через
useEffect при изменении `history.length` — **не использует rAF**, потому
что обновление редкое (раз в 5 тиков).

**Sparkline** — мини-линия одного метрика. Используется в per-player
карточке для показа траектории территории. 300×28 px.

Я думал про подключение библиотеки (recharts, chart.js) и решил **не**.
Эти две задачи решаются 200 строк ручного канваса, а библиотека добавила
бы 100+ КБ в bundle. На графиках чем меньше зависимостей — тем меньше
неожиданностей в production.

Smoke-тесты:
- 6 пресетов по 100 тиков — 6/6 PASS
- Unlimited stress: 500 тиков за 95 мс, 98 муравьёв в cap 2499 ✓
- GC integrity после 200 тиков — массив очищен ✓
- Большая симуляция 1000 тиков за 72 мс ✓

13/13 критериев приёмки из ТЗ §12 пройдены. Этап 2 закрыт.

Заметка для будущей главы книги: **в Этапе 2 был перелом в восприятии
проекта**. Раньше песочница ощущалась как "сделать настройки и смотреть".
После day/night и live-statistics — стала "запустить эксперимент и
анализировать". Тот же код, но воспринимается иначе. Видение пользователя
изменилось не от добавления функций, а от того что **существующая
функциональность стала видимой**. Это редкая ситуация когда product
value добавляется визуализацией без новой логики.

## Неделя 11 · Второе крупное переосмысление

### День 41 — Долой случайность

После закрытия Этапа 2 я предложил тимлиду начать Этап 3 как был в плане:
Event Log + Heatmaps + Highlights. Тимлид остановил:

> также добавить шаги назад тем самым давать возможность анализировать
> ситуацию и изучать те или иные моменты, и что не маловажно уже пора
> исключить все шансы и вероятности, только чистые правила и математика.

И это попало в точку.

В DEVLOG за День 35 я сам же написал: «Math.random() в клеточном автомате —
это запах архитектуры». Но **в коде** случайность оставалась — в выборе
родителя для рождения, в выборе клетки куда поставить ребёнка, в hybrid/
wild решении, в направлении нового муравья.

Это была **декларация без исполнения**. Я говорил одно а оставлял другое.

Тимлид не помнил мою фразу про «запах архитектуры» — он пришёл к тому же
выводу независимо, по причинам **геймдизайнерским**. Без устранения
случайности — невозможно отличить «стратегия победила» от «seed повезло».
А значит **песочница как тренажёр PvP** не работает.

Это второе крупное переосмысление в проекте. Первое — DEVLOG День 36,
когда мы поняли что песочница это **лаборатория для PvP правил**, а не
косметика. Второе — сейчас, когда мы поняли что **лаборатория требует
исключения случайности**. Иначе она не лаборатория, а казино с цветным
выводом.

Также тимлид настоял на **step forward / step back с настраиваемым шагом**.
Это вторая half фундаментальная вещь — без управления временем
исследование стратегии невозможно. Я предложил step controls с
preset-значениями (1, 5, 100). Тимлид поправил: «два по умолчанию а
третий адаптивный где мы сами можем прописать количество шагов». Лучше.
В научных тулзах это стандарт — pre-defined steps + custom field.

План пересмотрен:

- **Этап 3 — Determinism & Time Control** (раньше планировался Analytics)
- **Этап 4 — Analytics & Insights** (переехал)
- **Этап 5 — Mutation Lab** (без изменений)
- **Этап 6 — Sharing & Replay** (переехал)

Это правильный порядок. Detrandomization влияет на движок. Дальнейшие
этапы работают с детерминированным движком — проще писать тесты, проще
делать replay, проще объяснять поведение игроку.

По конкретным правилам замены случайности тимлид сказал умную вещь:

> тут нет возможности дать однозначный ответ, нужно экспериментировать

Это нормально. Когда ты убираешь случайность — на её место нужно ставить
правило. Какое правило **правильное** — узнаешь только запустив симуляцию.
Я выбрал свои дефолты (родитель = «окружённый», клетка = «первая свободная
по часовой стрелке», направление = «как у родителя»). Если они окажутся
плохими в эксперименте — поменяем. Это та же логика как с win conditions
в Mutation Lab: **сначала наблюдение, потом конкретное правило.**

Запись для будущей главы книги: **в этой точке проект окончательно
определил себя как "лаборатория для генерации правил PvP-игры"**. Не
"игра в которую можно играть". Игра — это сжатый результат лаборатории.
Эта переориентация повлияет на каждое решение дальше.

## Долгосрочное · Что я знаю сейчас, чего не знал в начале

**Про код:**
- Babel-standalone — отличный способ начать, ужасный способ продолжать
- TypeScript строгий режим (`noUncheckedIndexedAccess`) ловит реальные баги
- Детерминированная симуляция = маленькие реплеи + проверяемая логика
- Свой код "core/" между фронтом и бэком — не оптимизация, а необходимость
- `Math.random()` в клеточном автомате — это **запах архитектуры**, лучше детерминированное условие

**Про процесс:**
- Code review сразу после написания (день в день) ловит баги дёшево
- "Работает" ≠ "удовлетворяет требования". Лучше ТЗ на бумаге чем демо в голове
- Регрессионные тесты — это не для "галочки", они реально спасают (антипаттерн с tx() в v6)
- Один спринт ≠ переписать всё. Раздели на этапы, закрой первый полностью, потом следующий
- **Час обсуждения экономит день переделки.** Это не агильный мем, это эмпирика.

**Про людей:**
- Управление через страх — короткий выигрыш, долгий проигрыш
- Команда лучшего среднего > жёсткая команда из выживших
- Open-source с равными долями работает лучше чем "я босс, делайте"
- Признать ошибку вслух дороже чем удерживать позицию
- Тимлид, который не боится сказать "я был неправ" — даёт команде безопасность ошибаться. Это снимает 80% стресса разработки

**Про продукт:**
- Песочница не должна быть **косметикой к матчу**. Она должна быть **первичной**
- Лучшие game design идеи рождаются из **наблюдения за системой**, не из спецификации
- Open-source имеет смысл когда комьюнити может **расширять механики**, а не только фиксить баги
